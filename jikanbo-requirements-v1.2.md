## 1. プロダクトビジョン

> **「家計簿と同じことを、時間でやる」**

家計簿の本質的な価値は「毎月の固定費を把握したうえで可処分所得を明確にし、その使い道を計画することで、未来の資産状況をロードマップとして見通せること」にある。時間簿はこれを時間に適用する。

- 睡眠・経常業務などの **固定費** を先に確定する
- 残った **可処分時間** を可視化する
- 可処分時間に **重要な目標・ガントチャート** を割り当てる
- 朝の時点で「今日何をするか」が **自動的に** 決まっている状態をつくる
- 時間の「皮算用」から脱却する
- **使った時間を振り返り、見積もりのズレから学習する**（v1.1で強化）

## 3. コアモデル：時間の家計簿

```
週の総時間（168h）
  └─ 睡眠・食事・通勤 などの生活固定費　→ 除外
      └─ 稼働可能時間（収入）
            ├─ 固定費 … 会議・定例など「動かせない予定」
            ├─ 経常費 … 繰り返し発生する業務（週次・月次）
            ├─ 投資費 … 開発・学習など先取りしたい時間
            └─ 可処分時間（余剰）← ここに「目標」を割り当てる
```

### 季節変動（重要）
繁忙期と閑散期では稼働可能時間が変わる。**期間ごとに稼働時間を変えられること**が本質。さらに後述（§8）の通り、**分析の閾値も期間ごとに変えられる**。

```
閑散期：週40h稼働 → 可処分大
繁忙期：週28h稼働 → 可処分小 → 目標の詰め込みすぎを事前に検知
```

## 4. UIの方向性：クリップボード型（v1.1で確定）

### 基本構造：固定盤面＋文脈ページャ
画面を「タブで切り替える」のではなく、**盤面（レイアウト）は常に固定**で、**めくるのは"文脈"**とする。クリップボードの紙を一枚ずつめくるイメージ。

```
┌─[クリップ]──────────────┐
│  ◀  文脈名（例：顧客A大学）  ▶  │
│  ┌────────────────────┐  │
│  │   今月カレンダー（上半分）   │  │
│  ├──────────┬─────────┤  │
│  │  今週     │   年間    │  │
│  ├──────────┴─────────┤  │
│  │  4象限（この文脈のタスク／アイコン付）│  │
│  ├────────────────────┤  │
│  │   時間収支バー（最下部）     │  │
│  └────────────────────┘  │
│        ● ○ ○ ○（ページ）      │
└──────────────────────────┘
```

- ページをめくる（左右スワイプ／矢印／ドット）と、盤面全体がその文脈で再描画される。
- 文脈は **全予定／顧客別／プロジェクト別／プライベート** など（§5-8、可変・〜10枚）。
- 「全予定」ページは常に全文脈の**合算**を映す。

### 実装プラットフォームの方針（段階方式）
1. **Web（PWA）でプロトタイプ**：操作感の検証。→ **済**（`jikanbo-clipboard-mock.html`）。現行の機能版（`jikanbo.html`）の資産も活用。
2. **SwiftUIでネイティブ化**：紙をめくる質感、ウィジェット、Apple Watch、Vision Pro。→ 骨組み着手済（`ClipboardBoard.swift`）。
3. **EventKitで本連携**：Outlook/Appleカレンダーとの直接同期、朝のブリーフィングのプッシュ通知。→ 将来。

> ICS書き出し（§5-4, §5-5）は、EventKit直結が入るまでの主要な共有・連携手段。当面の最優先で維持する。

### 5-5.【新規】共有用ICSと公開レベル
| # | 要件 |
|---|---|
| R25 | **予定ごとの公開レベル（3段階）**：`公開`（詳細まで出す）／`多忙のみ`（タイトルを「予定あり」等に置換し場所・詳細を伏せる）／`非公開`（共有用には出力しない） |
| R26 | **2つの書き出しモード**：`自分用`（全件・全詳細）／`共有用`（公開＋多忙マスクのみ。非公開は出力しない） |

**設計上の注意（重要）**：iCalendar（RFC 5545）の `CLASS:PRIVATE` 等の属性は秘匿の保証にならない（取り込み側に中身のデータは渡るため）。確実に隠すには、**共有用モードでは非公開予定を出力しない／タイトル差し替え＋詳細削除**で実装する。`CLASS` 属性は補助的にのみ付与。

### 5-6.【新規】柔軟な予定入力と分析
**入力方針**：自由度を無限にせず、“少数の属性の掛け合わせ”で何でも表現できる状態を狙う。

| # | 要件 |
|---|---|
| R27' | **統一クイック入力**：種別ごとにフォームを分けず、1つの入力からどの型にもできる。属性＝カテゴリ（固定/定期/投資）× タグ × 文脈 × 重要度 × 緊急度 × 見積もり時間 × 公開レベル |
| R28 | **分析ダッシュボード**：①カテゴリ配分 ②象限配分（Q1〜Q4） ③タグ別（何に時間が溶けているか） ④文脈別 ⑤週次トレンド |
| R29 | **無駄の検出と削減候補**：Q3・Q4で時間を消費している項目を抽出し「削減候補リスト」を自動生成 |
| R35 | **カテゴリ・タグはアイコンで表現**（文字ラベルは任意・既定では付けない）。詳細は §5-9 |

**「何に集中すべき・何が無駄か」**は主に②象限配分と③タグ別で答えが出る。無駄の判定基準は §9。

| R33 | **見積補正係数**：上記の癖から、次回の見積もりに自動で補正をかける |

| # | 要件 |
|---|---|
| R34 | **文脈ページの可変管理**：追加・削除・並べ替えが可能（上限の目安 **〜10枚**）。現行3つ（顧客A/顧客B/プライベート）は固定で少なすぎるため撤廃 |

### 5-9.【新規】アイコンによるカテゴリ・タグ表現
**思想**：文字ラベルは意味（ニュアンス）を固定してしまう。あえて意味を“おぼろげ”に残し、各アイコンが自分にとって何を指すかは**ユーザーが決め、後から変えられる**ようにする。情報カードやノートに記号で意味を後付けしていく感覚に近い。

| # | 要件 |
|---|---|
| R35 | **カテゴリ・タグをアイコンで表示**。画面上は文字ラベルを既定では出さない |
| R35' | **アイコンは規定セット（記号10〜15個程度）から選択**。例：◇ ○ △ ☆ □ ◎ ● ▲ ■ ◆ ▽ ＋ … 。自由入力（任意の絵文字等）は**不可**（端末差での表示崩れ回避・見た目の統一） |
| R36 | **意味づけはユーザーに委ねる**。アイコンに既定の名前を付けない。補足ラベルは任意（空でも可）。ホバー／ロングタップで補足を出す程度に留める |

**設計上の注意**：カテゴリ（固定/定期/投資）は収支計算の根幹なので、**内部的には3区分の意味を保持**する（「投資」を可処分から先取りする等のロジックは不変）。アイコン化するのは**画面表示のみ**で、文字を消す。タグはもともと自由なので、アイコン＋色で運用し文字は任意。

## 6. 予定の統一データモデル（①②③を支える共通の器）
3つの新機能はバラバラではなく、**1つの予定に共通の属性を持たせる**ことで噛み合う。

```
予定（event / task）共通属性：
  title          … 名前
  contextId      … どの文脈（クリップボードのページ）に属するか
  category       … fixed | recurring | invest（内部区分。画面表示はアイコン。§5-9）
  tags[]         … タグID参照（各タグ＝アイコン＋色＋任意ラベル。§5-9）
  importance     … 重要度（★）
  urgency        … 緊急度（⚡）
  estimate       … 見積もり時間（分）
  actual         … 実績時間（分・終了後に記録）
  visibility     … public | busy | private（公開レベル）
  date / time    … 日付・時刻
```

- **① 共有ICS** は `visibility` で出し分ける。
- **② 分析** は `tags / importance / urgency / category / contextId` で集計する。
- **③ 見積もり精度** は `estimate` と `actual` を比べる。

→ 同じ器の別の使い方。新機能を足すときも、この属性セットに乗せる。

## 8.【新規】Q3過多＝業務改善のサイン（閾値可変・期間別）

「無駄の検出（§9）」とは**別軸**。Q3（重要でない×緊急）が多い状態は、個人の時間管理ではなく**業務側の構造**を見直すべき兆候。

| # | 要件 |
|---|---|
| R30 | **Q3過多の検知**：Q3が一定割合を超える、または特定タグのQ3が繰り返し溜まったら警告 |
| R30' | **改善候補の提示**：単に「無駄」と切らず、(1)定型化・テンプレ化できないか (2)委任・分担できないか (3)発生源（突発の問い合わせ等）を前段で抑えられないか、を提案 |
| R31 | **閾値はユーザーが都度変更可能**。固定値にしない |
| R31' | **閾値は期間（季節）ごとに設定可能**。例：通常期はQ3が可処分の20%超で警告、繁忙期は35%まで許容 |

**意図**：繁忙期は構造的にQ3が増えるため、固定閾値だと警告が鳴りっぱなしで無意味になる。期間別＋可変にすることで「繁忙期だから今は仕方ない」と「平常時なのにQ3が多い＝本当に見直すべき」を切り分ける。
（既定値は用意し、いつでも上書き可能とする）

## 9. 分析の基準（「無駄」の定義）

- **主軸：象限モデル**。Q3（緊急だが非重要）・Q4（どちらでもない）に費やした時間を「無駄寄り」と判定。
- **補助軸：目標寄与**。その時間が年間目標（§5-3）に効いているかを併せて見る二段構え。
- Q3については §8 の通り、無駄判定とは別に「業務改善のサイン」としても扱う。

## 10. データ構造（拡張後の想定）

```javascript
state = {
  activeContext: "ctx_all",       // 現在開いている文脈ページ
  contexts: [                     // R34：可変・〜10枚
    { id:"ctx_all",  name:"全予定",   type:"all",      color:null },
    { id:"ctx_a",    name:"顧客A大学", type:"customer", color:"#5B7186" },
    { id:"ctx_priv", name:"プライベート", type:"private", color:"#A9854F" },
    // … 追加・削除・並べ替え可
  ],
  iconSet: ["◇","○","△","☆","□","◎","●","▲","■","◆","▽","＋"], // 規定セット（10〜15個・固定）
  categoryIcons: { fixed:"◇", recurring:"○", invest:"☆" },        // ユーザーが割当・変更可
  tags: [ {id, icon, color, label} ],   // label は任意（空可）。icon は iconSet から選択
  periods: [                      // R02/R31'：季節変動
    { id, name:"通常期", weeklyIncome:40, q3ThresholdPct:20,
      start:null, end:null },
    { id, name:"繁忙期", weeklyIncome:28, q3ThresholdPct:35,
      start:"2026-08-20", end:"2026-09-10" },
  ],
  settings: { defaultVisibility:{ work:"public", private:"private" },
              estimateCorrection:true },
  books: {
    // 既存の work/private は contexts へ統合移行する想定。
    // 各予定は §6 の統一属性（contextId, estimate, actual, visibility, tags, importance, urgency …）を持つ。
  }
}
```

## 12. 残課題・保留事項
- **Outlook自動連携**：mailto下書きは可能だが直接送信は不可。EventKit（ネイティブ）かPower Automate等が必要。まず mailto / ICS で十分かを確認。
- **公開レベルの確実性**：ICSでの秘匿は保証されない。共有用は「非公開を出力しない／タイトル差し替え」で担保（§5-5）。
- **実績記録の継続性**：記録の手間が最大の障壁。ワンタップ or 自動推定を最優先。
- **データの永続化・同期**：現在 localStorage のみ。複数デバイス利用や端末間同期が必要になれば外部ストレージを検討。設定とデータのJSON書き出し機能を早めに用意するのが安全。

## 13. パフォーマンス方針（v1.2で明記）

**結論：個人利用の規模では重くならない。** ただし“ある作法”だけは最初から守る。

### データ量の見積もり
- 1予定 ≈ 数百バイト程度のテキスト。
- 仮に1日10件を5年ためても約18,000件＝数MB。これは localStorage の一般的上限（5〜10MB）内に収まり、JavaScript がこの件数を扱うのも一瞬。**体感が変わる水準ではない。**

### 重くなりうるのは「件数」ではなく「描画」
- 年間タイムラインに数千件のピンを一度に全部描く、カレンダーで全期間を毎回再計算する、といった作りにすると、件数が増えたとき表示がもたつく。

### 原則（実装で守る）
- **表示・集計は「見えている月・週・文脈の範囲」に限定する。全データを一度に描画・計算しない。**
- 分析（集計）も対象期間に絞る。必要なら事前集計をキャッシュし、データ変更時のみ再計算する。
- 大量描画が避けられない箇所は仮想化（見える範囲だけDOM生成）を検討。
